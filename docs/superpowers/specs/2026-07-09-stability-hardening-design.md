# 稳定性增强更新设计

## 背景

当前项目的核心主链路已经具备可用性：

- 登录、退出、再次登录已修复
- 主页持仓支持卖出与加仓
- 提醒系统支持 `above`、`below`、`reach`
- 后端核心回归测试已覆盖登录、加仓、提醒等主路径

现阶段最主要的风险不再是“缺少新功能”，而是：

- 前端构建缺少稳定验证
- 仓库缺少统一的自动化校验入口
- 上线前缺少清晰的冒烟检查清单
- 核心链路虽然已有测试，但缺少持续执行与可见的失败反馈
- 出现线上问题时，日志与排查入口仍偏弱

因此下一轮更新应优先聚焦稳定性，而不是继续扩展业务功能。

## 目标

本轮更新目标是为现有功能建立“上线前保护层”，让后续新增功能不会轻易破坏已有主链路。

成功标准：

- 每次推送代码都能自动检查后端测试与前端构建
- 本地和协作环境都能明确知道如何安装、构建和验证前端
- 有一份可执行的核心业务冒烟清单
- 关键写操作和提醒触发具备最基础的日志可观测性
- 当前已修复的关键边界问题有持续回归保护

## 范围

本次改动包含：

- 为仓库新增 GitHub Actions 自动校验工作流
- 固化前端运行与构建环境要求
- 更新 README 中的安装、构建、验证说明
- 补充核心业务冒烟测试清单文档
- 为关键后端业务链路补充最基础日志
- 对已有后端测试与边界校验做少量增强

本次改动不包含：

- 新增资产详情页
- 新增收益曲线或图表
- 新增邮件/IM 推送提醒
- 新增数据库 schema 变更
- 大规模重构前后端架构

## 方案对比

### 方案 A：稳定性优先的最小闭环增强

推荐方案。

做法：

- 加 GitHub Actions
- 固化前端环境与构建说明
- 补冒烟清单
- 补关键日志
- 补少量高价值回归测试

优点：

- 改动范围可控
- 风险低
- 立刻提升后续开发安全性
- 不会阻塞下一轮功能开发

缺点：

- 用户可见的新功能较少
- 主要收益体现在维护效率和回归控制

### 方案 B：稳定性与新功能并行推进

做法：

- 在做 CI 和验证的同时，继续开发资产详情页、收益曲线等新功能

优点：

- 用户能更快看到新功能

缺点：

- 范围容易膨胀
- 还没建立保护层时继续扩功能，回归风险更高
- 排障成本会上升

### 方案 C：只补文档，不补自动化

做法：

- 只更新 README 和手工检查清单
- 不做 GitHub Actions 和日志补强

优点：

- 实现最快

缺点：

- 无法自动防止回归
- 仍然依赖人工记忆执行验证
- 长期收益最小

## 设计结论

采用方案 A：先建立一层轻量但有效的稳定性保护，再进入下一轮功能开发。

本轮更新拆为 5 个子目标：

- GitHub Actions 自动校验
- 前端环境与构建固化
- 核心链路冒烟清单
- 核心后端日志增强
- 回归测试与边界保护补强

## 设计概览

### 1. GitHub Actions 自动校验

新增工作流，至少覆盖以下检查：

- 后端依赖安装
- 后端测试执行
- Python 语法编译检查
- 前端依赖安装
- 前端构建

建议触发时机：

- `push`
- `pull_request`

建议结构：

- `backend-check` job
- `frontend-build` job

这样可以把失败原因直接分成“后端失败”或“前端失败”，排查更快。

### 2. 前端环境与构建固化

当前仓库中前端构建依赖 Node/npm，但环境要求没有被足够明确地固定下来。

本轮需要补齐：

- 明确建议 Node 版本
- 明确 npm 安装命令
- 明确前端构建命令
- 明确前端构建失败时的排查路径

建议至少落地为：

- 更新 `README.md`
- 如项目已有前端目录配置空间，可增加版本说明文件，例如 `.nvmrc`

核心目标不是引入复杂工具链，而是减少“别人拉代码后前端跑不起来”的概率。

### 3. 核心链路冒烟清单

新增一份面向人工验证的冒烟清单文档，覆盖当前最关键的用户路径。

建议清单内容：

- 登录成功
- 退出后再次登录成功
- 新增资产成功
- 加仓成功
- 部分卖出成功
- 清仓卖出成功
- 创建 `above` 提醒成功
- 创建 `below` 提醒成功
- 创建 `reach` 提醒成功
- 提醒触发后前端展示正确
- 删除资产后关联提醒被清理

这份清单不替代自动化测试，但能为上线前验收提供统一脚本。

### 4. 后端关键日志增强

在不引入复杂监控系统的前提下，补一层轻量日志。

建议记录这些事件：

- 用户登录成功/失败
- 资产新增
- 资产加仓
- 资产卖出
- 资产删除
- 提醒创建/更新/删除
- 提醒触发

日志内容保持克制，只保留排查必要字段：

- 接口名
- 用户 ID
- 资产 ID 或提醒 ID
- 结果状态

不要记录敏感明文信息，例如密码、原始密钥内容。

### 5. 回归测试与边界保护补强

本轮不追求大规模新增测试，而是补“最容易再次出问题”的边界。

优先补强方向：

- GitHub Actions 中执行现有回归测试集
- 将当前已修复的非法 JSON 输入、提醒白名单、快照失效、提醒清理等路径持续纳入验证
- 如有必要，再补少量遗漏的关键组合用例

重点原则：

- 优先高价值回归
- 避免添加只重复实现逻辑的低价值测试

## 文件级设计

### 仓库级

建议新增或修改：

- `.github/workflows/ci.yml`
- `README.md`
- `docs/` 下的冒烟验证文档

### 后端

重点涉及：

- `backend/routes/auth.py`
- `backend/routes/assets.py`
- `backend/routes/alerts.py`
- `backend/routes/prices.py`
- 现有测试文件

改动方向：

- 加轻量日志
- 维持当前边界校验
- 让 CI 可直接执行现有测试命令

### 前端

重点涉及：

- `frontend/package.json`
- 可能新增 `.nvmrc`
- 必要时更新 README 中的前端构建说明

本轮不计划新增前端页面功能，仅围绕构建与环境一致性补强。

## 数据与兼容性

本轮不做数据库 schema 改动。

兼容性要求：

- 不修改真实数据库内容
- 不要求历史数据迁移
- 不改变现有 API 的业务语义
- 若新增日志，不影响现有响应结构

## 错误处理

### CI 失败

需要让失败信息可读：

- 后端测试失败时直接显示失败测试名
- 前端构建失败时直接显示构建命令输出

### 本地环境不一致

README 需要明确：

- 前端未安装依赖怎么办
- Node 版本不匹配怎么办
- 后端依赖如何安装

### 日志过量

日志增强应避免刷屏：

- 正常轮询类接口不额外大量打日志
- 仅对关键写操作和关键状态变更记录日志

## 验证方案

### 自动化验证

GitHub Actions 至少执行：

- `backend`: `./venv/bin/python -m unittest ...` 或等效统一命令
- `backend`: `./venv/bin/python -m compileall ...`
- `frontend`: `npm install`
- `frontend`: `npm run build`

### 手工验证

按冒烟清单逐项执行。

### 通过标准

满足以下条件即可认为本轮完成：

- GitHub Actions 工作流可运行
- README 能指导新环境完成安装和构建
- 冒烟清单文档已存在
- 后端关键链路具备最小日志能力
- 现有核心测试在自动流程中可执行

## 风险与取舍

### 风险 1：前端环境仍可能因外部依赖差异失败

处理方式：

- 先固定版本与命令
- 再通过 CI 统一收敛环境差异

### 风险 2：日志补太多导致噪声上升

处理方式：

- 只记录关键写操作
- 不给高频轮询加详细日志

### 风险 3：测试集持续增长导致 CI 变慢

处理方式：

- 当前只保留核心回归
- 后续再按需要拆分 smoke / full 两级测试

## 后续路线

本轮稳定性增强完成后，下一轮再进入更适合用户可见收益的功能迭代。

建议下一轮优先级：

- 资产详情页
- 交易明细与成本拆分
- 收益曲线
- 提醒高级策略，例如重复提醒和容差配置

本轮不提前把这些内容混入实现，避免范围膨胀。
