# 主页加仓与到达提醒设计

## 背景

当前主页持仓卡片仅支持卖出，不支持对已有持仓直接加仓。提醒系统当前仅支持“超过”和“低于”两种提醒类型，不支持“到达”。

本次需求目标是在不修改数据库结构的前提下：

- 在主页持仓卡片中增加“加仓”入口
- 对已有持仓执行加仓，并自动更新持仓数量与加权平均成本价
- 在提醒类型中增加“到达”
- 保持交易历史、主页持仓、提醒列表和触发文案的一致性

## 范围

本次改动包含：

- 主页持仓卡片新增“加仓”按钮与加仓弹窗
- 后端新增已有持仓的加仓接口
- 交易历史支持记录加仓行为，沿用 `buy` 动作
- 提醒类型新增 `reach`
- 提醒创建、编辑、展示和触发逻辑支持 `reach`
- 为加仓与 `reach` 提醒补充后端回归测试

本次改动不包含：

- 数据库 schema 变更
- 历史数据迁移
- 改动真实数据库内容以做手工修复
- 新增提醒类型之外的提醒策略配置

## 方案对比

### 方案 A：专用加仓接口 + 到达容差触发

推荐方案。

- 加仓使用专用接口处理数量、金额、成本价更新和交易历史记录
- 到达提醒新增独立类型 `reach`
- 价格触发时按小容差判断“已到达目标价”，避免浮点误差导致无法触发

优点：

- 业务语义清晰
- 交易历史完整
- 主页与资产页行为一致
- 提醒触发更贴近实际使用

### 方案 B：复用资产编辑接口 + 到达严格等于触发

- 前端直接改持仓数量和成本价
- 提醒仅在当前价严格等于目标价时触发

缺点：

- 缺失加仓交易历史
- 需要用户手工算新成本价
- 严格等于在实际行情里不稳定

### 方案 C：加仓时新增一条独立资产记录 + 到达映射为超过或低于

- 不改已有持仓，只新增另一条资产
- 到达提醒不做真实第三种类型

缺点：

- 会把同一持仓拆成多条
- 主页面不符合“对现有持仓加仓”的预期
- 提醒含义不清晰

## 设计概览

### 主页加仓

- 在 `Dashboard.tsx` 的持仓卡片中，将右侧操作区从单个“卖出”按钮调整为“加仓”与“卖出”两个按钮
- 点击“加仓”打开新的弹窗
- 弹窗字段：
  - 加仓价
  - 数量
  - 或金额
- 用户填写数量或金额其中之一即可提交

### 加仓成本规则

加仓后的成本价按加权平均计算：

`新成本价 = (原成本价 * 原数量 + 加仓价 * 加仓数量) / 新总数量`

同时更新：

- `quantity` 为新总数量
- `buy_price` 为新成本价

### 加仓交易历史

- 加仓行为写入 `TradeRecord`
- 动作继续使用现有 `buy`
- 历史列表中显示为“买入”
- 不新增新的数据库字段或枚举值

这样可以避免数据库结构改动，同时保持已有“买入/卖出”展示逻辑可复用。

### 到达提醒

- 提醒类型新增 `reach`
- 前端下拉文案新增“价格到达...”
- 列表文案和主页触发提示支持 `reach`

`reach` 的触发逻辑不是“严格等于”，而是“当前价进入目标价的容差范围”

推荐容差规则：

- 使用 `0.001` 作为基础容差
- 触发条件为 `abs(current_price - target_price) <= 0.001`

这样可以避免浮点数比较导致的不可触发问题，同时实现成本较低。如后续需要按资产类型细化 tick 规则，再单独扩展。

## 前端设计

## 后端设计

### 加仓接口

涉及文件：

- `backend/routes/assets.py`

新增接口：

- `POST /api/assets/<asset_id>/add-position`

请求体：

- `buy_price`：必填，必须大于 0
- `quantity`：选填
- `amount`：选填

校验规则：

- 资产必须属于当前用户
- `buy_price` 必须大于 0
- `quantity` 和 `amount` 至少提供一个
- 若仅提供 `amount`，则按 `amount / buy_price` 推导数量
- 推导后的 `quantity` 必须大于 0

处理逻辑：

1. 读取原持仓数量和原成本价
2. 计算本次加仓数量
3. 按加权平均计算新成本价
4. 更新资产数量与成本价
5. 写入一条 `buy` 类型交易历史
6. 提交事务

返回值：

- 成功消息
- 更新后的资产摘要
- 本次加仓数量

### 到达提醒创建与编辑

涉及文件：

- `backend/routes/alerts.py`

改动点：

- 创建提醒时允许 `alert_type = reach`
- 编辑提醒时允许 `alert_type = reach`
- 不新增数据库字段

### 到达提醒触发

涉及文件：

- `backend/routes/prices.py`

改动点：

- 持仓提醒与手动提醒的检查逻辑都支持 `reach`
- 触发条件扩展为：
  - `above`: `current_price >= target_price`
  - `below`: `current_price <= target_price`
  - `reach`: `abs(current_price - target_price) <= 0.001`

触发后行为保持不变：

- 标记 `triggered = True`
- 写入 `triggered_at`
- 返回触发提醒列表给前端展示

## 数据流

### 加仓数据流

1. 用户在主页点击“加仓”
2. 前端打开加仓弹窗并填写价格与数量/金额
3. 前端调用 `POST /api/assets/<asset_id>/add-position`
4. 后端更新持仓数量与成本价，并记录 `buy` 历史
5. 前端刷新：
  - 主页组合
  - 交易历史
6. 前端切换到历史 tab，让用户看到最新记录

### 到达提醒数据流

1. 用户在提醒页创建或编辑 `reach` 类型提醒
2. 前端提交提醒配置
3. 后端保存 `alert_type = reach`
4. 定时价格检查时执行 `reach` 触发判断
5. 命中后返回触发提醒
6. 前端以“到达目标价”文案展示

## 错误处理

### 加仓

- 资产不存在：返回 404
- 价格为空或小于等于 0：返回 400
- 数量和金额都为空：返回 400
- 推导后数量小于等于 0：返回 400

前端提示文案沿用现有弹窗 `alert(error.message || '加仓失败')`

### 提醒

- `alert_type` 非法：返回 400
- 目标价为空：返回 400
- 持仓提醒关联资产不存在：返回 404

## 测试策略

### 后端测试

新增或扩展测试，覆盖：

- 加仓后数量增加正确
- 加仓后成本价按加权平均更新
- 加仓后新增一条 `buy` 交易历史
- `reach` 提醒可成功创建
- `reach` 提醒在价格进入容差范围时会触发

### 前端回归重点

人工验证以下流程：

- 主页卡片同时显示“加仓”和“卖出”
- 加仓成功后主页仓位、成本价、市值和收益刷新
- 历史记录中出现新的买入记录
- 提醒类型下拉可选“到达”
- 列表和主页提醒横幅对 `reach` 文案显示正确

## 兼容性与约束

- 不修改数据库结构
- 不手工修改真实数据库内容
- 继续复用现有 `TradeRecord.action = buy/sell`
- 不改变现有卖出逻辑
- 不改变提醒的暂停、删除和通知方式逻辑

4. 修改主页持仓卡片与加仓弹窗
5. 修改提醒页类型选择与文案
6. 联调并回归验证主页与提醒页
